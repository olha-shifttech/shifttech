## 1. Назва документа



11_Docs_Storage_And_Sync_Rules.md



Статус: Draft



Версія: 1.0



Власник: ShiftTech



Призначення: зафіксувати, де зберігаються документи проєкту ShiftTech 2.0, які копії є головними, що дублюється в GitHub, що додається в ChatGPT Project Sources, і як не створювати хаос.

## 2. Навіщо потрібен цей документ



У проєкті ShiftTech 2.0 буде багато файлів:



- стратегічні документи;

- технічні ТЗ;

- контент-плани;

- шаблони;

- промпти для Codex;

- файли сайту;

- кейси ремонту;

- фото / відео;

- SEO-документи;

- Make-сценарії;

- API-документація.



Без правил зберігання швидко з'являться дублікати, плутанина і старі версії.



Тому потрібне правило: кожен файл має своє місце.



---



## 3. Головне правило



Google Drive — робоче сховище для бізнесу.



GitHub — технічне сховище для коду і документації сайту.



ChatGPT Project Sources — контекст для AI.



Codex — середовище виконання задач, а не головне сховище знань.



---



## 4. Ролі сховищ



### Google Drive



Використовується для:



- головних проєктних документів;

- робочих Google Docs;

- кейсів ремонту;

- фото і відео;

- накладних;

- актів;

- таблиць;

- матеріалів, які зручно переглядати команді.



### GitHub



Використовується для:



- коду сайту;

- markdown-документації;

- технічних правил для Codex;

- структури /docs;

- версійності технічних файлів;

- content schemas;

- компонентів сайту.



### ChatGPT Project Sources



Використовується для:



- ключових правил проєкту;

- архітектури;

- моделі даних;

- content rules;

- технічних вимог;

- документів, які AI повинен пам'ятати під час роботи.



### Codex



Використовується для:



- генерації коду;

- редагування репозиторію;

- запуску технічних задач;

- перевірки структури сайту.



Codex не повинен бути єдиним місцем зберігання знань.

## 19. Правило синхронізації



Коли документ змінюється, потрібно визначити, куди оновити копію.



### Якщо змінився стратегічний документ



Оновити:



- Google Drive;

- ChatGPT Project Sources;

- GitHub /docs, якщо це впливає на сайт або Codex.



### Якщо змінився технічний документ



Оновити:



- Google Drive;

- GitHub /docs;

- ChatGPT Project Sources, якщо AI має це пам'ятати.



### Якщо змінився контент сторінки



Оновити:



- GitHub content files або CMS / content folder;

- Google Drive draft, якщо він був джерелом;

- SEO / Knowledge Base нотатку, якщо змінено структуру.

## 21. Статуси документів



Дозволені статуси:



- Draft

- Active

- Needs Review

- Approved

- Deprecated

- Archived



Active означає: документ зараз є робочим правилом.



Deprecated означає: документ більше не є актуальним, але зберігається для історії.



Archived означає: документ перенесено в архів.



---



## 22. Правило іменування файлів



Файли повинні мати номер і зрозумілу назву.



Формат:



NN_File_Name.md



Приклади:



- 01_Project_Constitution.md

- 09_Data_Model.md

- 10_Content_Rules.md



Заборонено:



- file1

- new doc

- копія копія

- фінал 2

- документ без номера



---



## 23. Правило дублювання



Дублікати заборонені.



Якщо потрібно зробити копію, треба чітко вказати її роль:



- Google Drive working copy;

- GitHub technical copy;

- ChatGPT source copy;

- Archive copy.



Копія без ролі вважається хаосом.



---



## 24. Правило перед створенням нового документа



Перед створенням нового документа потрібно відповісти:



1. Для чого він потрібен?

2. Чи немає вже такого документа?

3. В яку папку він має йти?

4. Чи потрібен він у GitHub?

5. Чи потрібен він у ChatGPT Project Sources?

6. Чи впливає він на Codex?



---



## 25. Правило для Codex



Перед запуском Codex потрібно переконатися, що:



- актуальні документи є в GitHub /docs;

- Codex має доступ до документації;

- Codex не отримує застарілий промпт;

- 08_Codex_Build_Prompt.md відповідає 07_Technical_Site_Requirements.md;

- 09_Data_Model.md врахований;

- 10_Content_Rules.md врахований.



---



## 26. Правило для ChatGPT



Коли ChatGPT працює над проєктом ShiftTech, він повинен:



- спиратися на документи 01–11;

- не вигадувати архітектуру заново;

- не створювати нові правила без потреби;

- перевіряти, куди зберігається результат;

- попереджати про дублікати;

- говорити, що було змінено і де.



---



## 27. Мінімальний порядок роботи перед запуском Codex



1. Перевірити документи 01–11 у Google Drive.

2. Додати копії в GitHub /docs.

3. Додати ключові документи в ChatGPT Project Sources.

4. Перевірити, що 08_Codex_Build_Prompt.md актуальний.

5. Запустити Codex тільки після цього.



---



## 28. Що не робимо зараз



Не створюємо:



- складну систему версій;

- окрему CMS;

- базу даних документів;

- автоматичну синхронізацію Google Drive ↔ GitHub;

- зайві папки.



Спочатку достатньо ручної дисципліни і чітких правил.



---



## 29. Критерій правильності



Система зберігання працює правильно, якщо:



1. Кожен файл має місце.

2. Кожен важливий документ має номер.

3. Немає хаотичних копій.

4. Зрозуміло, яка копія головна.

5. Codex бере правила з GitHub /docs.

6. ChatGPT має правила в Project Sources.

7. Google Drive залишається зрозумілим для людини.



---



## 30. Коротка формула



Google Drive — для роботи і бізнесу.



GitHub — для коду і технічної документації.



ChatGPT Project Sources — для пам'яті AI.



Codex — для виконання задач.



Кожен файл має місце, роль і статус.
