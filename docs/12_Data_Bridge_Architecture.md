# 12_Data_Bridge_Architecture.md



**Project:** ShiftTech 2.0  

**Status:** Draft  

**Version:** 1.0  

**Owner:** ShiftTech  

**Purpose:** описати, як знання та дані проходять шлях від Google Drive до GitHub, Codex, сайту та майбутнього AI Search.



---



## 1. Мета документа



Цей документ фіксує Data Bridge Architecture для ShiftTech 2.0.



Data Bridge — це міст між:



- Google Drive;

- GitHub;

- Codex;

- сайтом;

- майбутнім AI Search;

- майбутніми AI-агентами.



Мета документа — не допустити хаосу, дублювання і втрати джерела правди під час перенесення знань у технічну структуру сайту.



---



## 2. Головна проблема



У ShiftTech 2.0 є кілька шарів даних:



1. Робочі документи у Google Drive.

2. Технічна документація у GitHub.

3. Код сайту.

4. Контент сайту.

5. Майбутня база для AI Search.

6. Майбутні автоматизації Make / API.



Якщо не зафіксувати правила переходу між цими шарами, з'являться:



- дублікати;

- старі версії;

- різні назви одних і тих самих сутностей;

- застарілий контент у Codex;

- хаотичні сторінки сайту;

- AI-відповіді на основі неправильних даних.



---



## 3. Головний принцип



Один факт має одне головне місце зберігання.



Інші місця можуть використовувати цей факт, але не повинні створювати нову незалежну версію.



Формула:



Source of Truth → Structured Data → Website → AI Search



---



## 4. Ролі основних систем



### 4.1. Google Drive



Google Drive — робоче бізнес-сховище.



Тут зберігаються:



- проєктні документи;

- архітектура;

- робочі чернетки;

- матеріали по ремонтах;

- фото і відео;

- накладні;

- акти;

- нотатки майстра;

- матеріали, які зручно переглядати команді.



Google Drive — місце, де знання народжуються і проходять первинну перевірку.



---



### 4.2. GitHub



GitHub — технічне джерело правди для сайту.



Тут зберігаються:



- markdown-документація;

- код сайту;

- технічні правила;

- структури даних;

- content files;

- schemas;

- компоненти;

- історія змін.



GitHub не замінює Google Drive для бізнес-документів, але є офіційним технічним середовищем для сайту.



---



### 4.3. Codex



Codex — виконавець технічних задач.



Codex не є джерелом правди.



Codex повинен:



- читати затверджені документи;

- працювати з GitHub;

- не вигадувати архітектуру;

- не створювати дублікати;

- не змінювати модель даних без дозволу;

- зупинятися, якщо джерело неясне.



---



### 4.4. Website



Сайт — публічний шар знань.



Він показує тільки перевірену або безпечно сформульовану інформацію.



Сайт не повинен містити:



- приватні дані клієнтів;

- неперевірені технічні твердження як факт;

- хаотичні сторінки без зв'язків;

- дублікати одних і тих самих фактів.



---



### 4.5. AI Search



AI Search — майбутній шар поверх структурованої бази знань.



AI Search повинен використовувати:



- структуровані сутності;

- зв'язки між авто, коробками, симптомами, помилками і кейсами;

- статус перевірки;

- джерела фактів;

- правила приватності.



AI Search не повинен відповідати з неперевірених чернеток.

## 6. Шари даних



### Layer 1 — Raw Materials



Сирі матеріали:



- фото;

- відео;

- голосові нотатки;

- документи ремонту;

- первинні записи;

- скріни діагностики;

- нотатки майстра.



Місце зберігання:



Google Drive.



Статус:



Draft / Raw / Needs review.



---



### Layer 2 — Reviewed Knowledge



Перевірені знання:



- висновки по кейсу;

- підтверджені симптоми;

- підтверджені помилки;

- підтверджені зв'язки авто → коробка;

- технічні спостереження.



Місце зберігання:



Google Drive / 01_Knowledge_Base.



Статус:



Reviewed / Needs verification / Confirmed.



---



### Layer 3 — Canonical Architecture



Канонічна архітектура:



- правила проєкту;

- модель даних;

- структура сайту;

- правила контенту;

- технічні вимоги;

- ADR.



Місце зберігання:



Google Drive / 00_Project_Management.



Після затвердження:



GitHub /docs.



---



### Layer 4 — Structured Content



Структурований контент для сайту:



- MDX;

- Markdown;

- JSON;

- TypeScript objects;

- frontmatter;

- schemas.



Місце зберігання:



GitHub.



Можливі папки:



- /content/cars

- /content/transmissions

- /content/symptoms

- /content/errors

- /content/cases

- /content/faq

- /lib/data

- /lib/schema



---



### Layer 5 — Published Website



Опублікований сайт:



- сторінки авто;

- сторінки коробок;

- сторінки симптомів;

- сторінки помилок;

- кейси;

- FAQ;

- статті;

- контакти.



Місце:



Website / Production.



---



### Layer 6 — AI Index



Майбутній AI-шар:



- embeddings;

- vector database;

- structured index;

- entity graph;

- search API;

- AI agent context.



Місце:



майбутня AI Search / API layer.

## 7. Правило статусів



Кожен матеріал повинен мати статус.



Рекомендовані статуси:



- Raw

- Draft

- Needs Review

- Needs Verification

- Confirmed

- Approved

- Published

- Archived



Контент не може перейти на сайт без статусу:



Approved



або



Confirmed with safe wording.

## 9. Як Google Docs перетворюються на сайт



### Step 1 — Збір матеріалу



Матеріал створюється у Google Drive.



Приклади:



- кейс ремонту;

- нотатка майстра;

- фото;

- технічний висновок;

- контент-план;

- чернетка сторінки.



---



### Step 2 — Перевірка



Матеріал перевіряється:



- власником бізнесу;

- майстром;

- AI-архітектором;

- технічним правилом needsVerification.



---



### Step 3 — Структурування



Матеріал перетворюється у формат:



- Vehicle Page;

- Transmission Page;

- Symptom Page;

- Error Code Page;

- Repair Case Page;

- FAQ.



Шаблон береться з:



05_Content_Templates.md



Правила беруться з:



10_Content_Rules.md



Модель даних береться з:



09_Data_Model.md



---



### Step 4 — Перенесення в GitHub



Після затвердження контент переноситься у GitHub.



Можливі формати:



- .md

- .mdx

- .json

- .ts



Codex не повинен переносити контент без чіткого джерела і статусу.



---



### Step 5 — Генерація сторінки



Next.js використовує структурований контент і генерує сторінки.



Сторінка повинна мати:



- правильний URL;

- H1;

- summary;

- entity facts;

- related links;

- FAQ;

- CTA;

- metadata;

- status / verification marker, якщо потрібно.



---



### Step 6 — Публікація



Публікується тільки контент, який не порушує правила приватності і не містить неперевірених тверджень як факт.



---



## 10. Правило для Codex



Codex повинен працювати за таким порядком:



1. Прочитати /docs.

2. Визначити, яке джерело є головним.

3. Не дублювати модель даних.

4. Не вигадувати факти.

5. Якщо потрібна структура сутності — брати її з 09_Data_Model.md.

6. Якщо потрібен шаблон сторінки — брати його з 05_Content_Templates.md.

7. Якщо потрібні правила тексту — брати їх з 10_Content_Rules.md.

8. Якщо потрібна технічна реалізація — брати її з 07_Technical_Site_Requirements.md.

9. Якщо є конфлікт між документами — зупинитися і повідомити.



---



## 11. Правило проти дублювання



Заборонено:



- дублювати повний опис полів сутностей у кількох документах;

- створювати різні slug для однієї сутності;

- створювати дві сторінки про одну й ту саму коробку;

- публікувати один кейс у двох різних форматах без зв'язку;

- копіювати Google Doc у GitHub без позначення статусу;

- створювати content file без джерела.



---



## 12. Майбутня автоматизація



У майбутньому Data Bridge може бути автоматизований:



Google Drive / Google Docs



↓



Make / API



↓



Validation



↓



GitHub Pull Request



↓



Codex Review



↓



Website Build



↓



AI Index Update



Це не обов'язково для MVP, але архітектура повинна не блокувати такий розвиток.



---



## 13. MVP-правило



На етапі MVP не потрібно будувати складну автоматизацію.



Достатньо:



1. Google Drive як робоче джерело.

2. GitHub як технічне джерело.

3. Ручне підтвердження перед переносом.

4. Codex як виконавець.

5. Чітке правило, який документ за що відповідає.



Головне — не створити хаос на старті.



---



## 14. Фінальний висновок



Data Bridge Architecture потрібна для того, щоб ShiftTech 2.0 могла масштабуватися без втрати логіки.



Головне рішення:



Google Drive — місце народження знань.



GitHub — технічне джерело правди для сайту.



Codex — виконавець.



Website — публічний шар.



AI Search — майбутній інтелектуальний шар.



Жоден шар не повинен хаотично дублювати інший.



Кожен факт повинен мати джерело, статус і правильний шлях до публікації.
