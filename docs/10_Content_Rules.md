# ShiftTech Content Rules



## 1. Назва документа



10_Content_Rules.md



Статус: Draft



Версія: 1.0



Власник: ShiftTech



Призначення: зафіксувати єдині правила створення, перевірки, структурування і публікації контенту для ShiftTech Platform.



---



## 2. Для чого потрібен цей документ



Цей документ потрібен, щоб усі сторінки ShiftTech були написані в одному стилі, за однією логікою і без хаосу.



Він застосовується до:



- сторінок автомобілів;

- сторінок коробок передач;

- сторінок симптомів;

- сторінок помилок;

- кейсів ремонту;

- FAQ;

- діагностичних гайдів;

- технічних статей;

- майбутнього AI Search;

- контенту для Codex;

- контенту для GitHub / docs.



---



## 3. Головний принцип контенту



Контент ShiftTech повинен бути корисним для людини, точним для майстра і зрозумілим для AI.



Кожна сторінка має відповідати трьом питанням:



1. Про що ця сторінка?

2. З якими сутностями вона пов'язана?

3. Яку дію має зробити клієнт після прочитання?



---



## 4. Основна формула



Автомобіль — точка входу для людини.



Коробка передач — центр моделі даних.



Симптом і помилка — міст між клієнтом і діагностикою.



Кейс ремонту — доказ експертності.



FAQ — коротка відповідь на конкретне питання.



AI Search — майбутній шар поверх правильно структурованого контенту.



---



## 5. Тон голосу ShiftTech



Контент повинен бути:



- простим;

- точним;

- спокійним;

- професійним;

- без залякування клієнта;

- без перебільшень;

- без порожнього SEO-тексту;

- без фраз, які звучать як реклама низької якості.



---



## 6. Що заборонено в текстах



Заборонено:



- вигадувати технічні факти;

- писати діагноз без діагностики;

- обіцяти точну причину без перевірки;

- писати “точно потрібно міняти”, якщо це не підтверджено;

- копіювати тексти конкурентів;

- створювати сторінки тільки заради SEO;

- дублювати один і той самий факт у різних місцях;

- використовувати різні назви для однієї сутності;

- змішувати кілька тем в одну сторінку;

- публікувати внутрішні дані клієнта без дозволу;

- публікувати VIN без перевірки, що це дозволено.



---



## 7. Правило технічної обережності



Якщо факт не підтверджений, потрібно писати обережно.



Правильно:



- може бути пов'язано з;

- часто зустрічається при;

- потребує діагностики;

- потрібно перевірити;

- одна з можливих причин;

- без діагностики точну причину визначити не можна.



Неправильно:



- причина точно в гідроблоці;

- потрібно міняти коробку;

- це 100% соленоїди;

- можна їхати без проблем;

- ремонт буде коштувати точно стільки-то без огляду.



---



## 8. Правило Source of Truth



Один факт — одне основне місце.



Приклади:



- технічна інформація про коробку зберігається в Transmission Page;

- інформація про авто зберігається у Vehicle Page;

- опис симптому зберігається у Symptom Page;

- код помилки зберігається в Error Code Page;

- ремонтна історія зберігається в Repair Case Page.



Інші сторінки повинні не дублювати факт, а посилатися на основну сторінку.



---



## 9. Обов'язкові блоки кожної сторінки



Кожна сторінка повинна мати:



- H1;

- коротке резюме;

- основну сутність;

- пов'язані сутності;

- внутрішні посилання;

- простий висновок;

- CTA;

- статус перевірки або позначку needsVerification, якщо потрібно.



---



## 10. Правило короткого резюме



На початку кожної сторінки має бути коротке резюме.



Резюме повинно пояснювати:



- про що сторінка;

- для кого вона;

- що важливо знати;

- коли потрібна діагностика;

- чим може допомогти ShiftTech.



Резюме не повинно бути довгим.



Оптимально: 3–6 речень.

## 12. Правило назв



Назви повинні бути стабільні.



Приклади правильних назв:



- Volkswagen Passat B7 2.5 USA

- Volkswagen Jetta 2.5 USA

- 09G

- JF010E

- JF011E

- JF015E

- JF016E

- JF017E

- P0700

- P0741

- P0965

- P1790



Не використовувати хаотично:



- 09 G

- JF 011

- джатко 11

- пассат америка

- коробка ниссан



У клієнтському тексті можна використовувати прості пояснення, але основна назва сутності має бути однакова.



---



## 13. Правило slug



Slug має бути стабільний і зрозумілий.



Правила:



- lowercase;

- латиниця;

- дефіси замість пробілів;

- без кирилиці;

- без випадкових скорочень;

- не змінювати після публікації без причини.



Приклади:



- /cars/volkswagen/passat-b7-2-5-usa

- /cars/volkswagen/jetta-2-5-usa

- /transmissions/09g

- /transmissions/jf010e

- /transmissions/jf011e

- /symptoms/jerks

- /errors/p0741

- /cases/st-2026-0001



---



## 14. Правило для Vehicle Page



Сторінка авто повинна бути написана для клієнта, який не знає коробку.



Обов'язково пояснити:



- яке це авто;

- який двигун / ринок важливий;

- які коробки можуть стояти;

- як зрозуміти, яка коробка ймовірна;

- які типові симптоми;

- які помилки можуть бути;

- коли потрібна діагностика;

- куди перейти далі.



Не робити сторінку авто просто SEO-текстом.



---



## 15. Правило для Transmission Page



Сторінка коробки передач — технічний центр знань.



Обов'язково:



- назва коробки;

- альтернативні назви;

- виробник;

- тип;

- сімейство;

- на яких авто зустрічається;

- типові симптоми;

- типові помилки;

- слабкі місця;

- пов'язані кейси;

- що перевірити перед ремонтом;

- що не міняти без діагностики.



---



## 16. Правило для Symptom Page



Сторінка симптому повинна перекладати мову клієнта в технічну логіку.



Обов'язково:



- як клієнт це відчуває;

- коли проявляється;

- на яких авто може зустрічатися;

- на яких коробках може зустрічатися;

- можливі причини;

- рівень ризику;

- чи можна їхати далі;

- що перевірити спочатку;

- пов'язані помилки;

- пов'язані кейси.



---



## 17. Правило для Error Code Page



Сторінка помилки повинна пояснювати код і не робити поспішних висновків.



Обов'язково:



- код;

- коротке пояснення для клієнта;

- технічне пояснення;

- на яких коробках зустрічається;

- на яких авто може зустрічатися;

- можливі причини;

- що перевірити спочатку;

- що не міняти без діагностики;

- пов'язані симптоми;

- пов'язані кейси.



---



## 18. Правило для Repair Case Page



Кейс ремонту — це доказ експертності, а не просто історія.



Обов'язково:



- номер кейсу;

- дата;

- авто;

- двигун, якщо відомо;

- коробка;

- пробіг;

- скарга клієнта;

- симптоми;

- помилки;

- діагностика;

- що було знайдено;

- що зроблено;

- які деталі замінено;

- результат;

- висновок для клієнта;

- висновок для бази знань;

- пов'язані сторінки.



Правило приватності:



VIN, номер авто, телефон, ПІБ клієнта та інші персональні дані не публікуються без дозволу.



---



## 19. Правило для FAQ



FAQ має відповідати на одне конкретне питання.



Структура:



- питання;

- коротка відповідь;

- уточнення, якщо потрібно;

- пов'язані сторінки;

- коли звертатися на діагностику.



FAQ не повинен перетворюватися на довгу статтю.



---



## 20. Правило для Diagnostic Guide



Діагностичний гайд — це алгоритм.



Структура:



1. Симптом

2. Авто

3. Ймовірна коробка

4. Помилки

5. Що перевірити без зняття коробки

6. Що перевірити глибше

7. Що не міняти одразу

8. Коли потрібна дефектовка

9. Висновок



---



## 21. Правило статусів контенту



Кожен матеріал має мати статус.



Статуси:



- draft

- needs_info

- needs_verification

- reviewed

- approved

- published

- needs_update

- archived



Контент без статусу не вважається керованим.



---



## 22. Правило needsVerification



Якщо є сумнів — ставимо needsVerification.



Це краще, ніж опублікувати неточний технічний факт.



Приклади:



- невідомий точний тип коробки;

- немає VIN;

- немає підтвердженої діагностики;

- майстер ще не підтвердив висновок;

- немає фото або дефектовки;

- причина помилки тільки припущення.



---



## 23. Правило для AI-friendly контенту



AI повинен легко зрозуміти сторінку.



Для цього потрібно:



- чіткий H1;

- коротке резюме;

- структуровані блоки;

- списки сутностей;

- однакова термінологія;

- внутрішні посилання;

- FAQ;

- висновок;

- відсутність води;

- явні зв'язки між авто, коробкою, симптомом, помилкою і кейсом.



---



## 24. Правило SEO



SEO важливе, але не головніше за користь.



Правильно:



- писати під реальний запит клієнта;

- використовувати чистий заголовок;

- додавати внутрішні посилання;

- відповідати на конкретне питання;

- робити сторінку корисною.



Неправильно:



- набивати ключові слова;

- писати однакові тексти з різними назвами авто;

- створювати сторінки без реальної користі;

- вигадувати симптоми тільки для трафіку.



---



## 25. Правило мови



Основна мова першої версії — українська.



Дозволено використовувати технічні англійські терміни, якщо вони є стандартними:



- CVT

- DSG

- TCM

- VIN

- Error Code

- Transmission

- Torque Converter



Технічні терміни потрібно пояснювати простими словами, якщо сторінка орієнтована на клієнта.



---



## 26. Правило доказовості



Краще менше тверджень, але точніших.



Джерела фактів:



- власний кейс ShiftTech;

- фото / відео ремонту;

- дефектовка;

- підтвердження майстра;

- технічний мануал;

- офіційна документація;

- перевірене зовнішнє джерело.



Якщо джерела немає — факт позначається як потребує перевірки.



---



## 27. Правило CTA



Кожна сторінка повинна мати наступну дію.



CTA може бути:



- запис на діагностику;

- надіслати VIN;

- надіслати коди помилок;

- надіслати відео симптомів;

- подзвонити;

- перейти до пов'язаної сторінки;

- прочитати кейс.



CTA має бути логічним, а не агресивним.



---



## 28. Перед публікацією перевірити



Перед публікацією сторінки потрібно перевірити:



1. Є H1.

2. Є коротке резюме.

3. Є головна сутність.

4. Є внутрішні посилання.

5. Є пов'язані авто / коробки / симптоми / помилки / кейси.

6. Немає непідтверджених тверджень як фактів.

7. Є статус.

8. Є CTA.

9. Немає дублювання.

10. Сторінка корисна людині.

11. Сторінка зрозуміла AI.

12. Є місце, де зберігається джерело правди.

## 30. Коротка формула



Не пишемо багато слабкого контенту.



Пишемо структуровані сторінки, які:



- допомагають клієнту;

- не вигадують технічні факти;

- пов'язані з моделлю даних;

- підсилюють експертність ShiftTech;

- зрозумілі пошуку та AI.
