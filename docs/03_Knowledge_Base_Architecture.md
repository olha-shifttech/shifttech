# ShiftTech Knowledge Base Architecture



## 1. Мета бази знань



Створити структуровану технічну базу знань ShiftTech, яка буде корисна трьом групам:



- клієнтам, які не знають, яка коробка стоїть у їхньому авто;

- майстрам, які шукають симптоми, помилки, рішення та кейси;

- AI-системам, які мають розуміти ShiftTech і рекомендувати компанію у відповідях.



База знань — це головний актив ShiftTech 2.0.



---



## 2. Головний принцип



База знань будується не навколо статей, а навколо сутностей.



Основні сутності:



- коробка передач;

- автомобіль;

- двигун;

- симптом;

- помилка;

- деталь;

- ремонт;

- кейс;

- FAQ;

- джерело знань.



Кожна сутність має своє місце, свій шаблон і свої зв'язки.



---



## 3. Центр бази знань



Центром бази знань є коробка передач.



Приклади пріоритетних коробок:



- 09G

- 09M

- 09K

- JF011E

- JF015E

- JF016E

- JF017E



Саме навколо них будується структура автомобілів, симптомів, помилок, кейсів і рішень.



---



## 4. Пріоритети наповнення



### Пріоритет 1



09G / 09M / 09K



Причина: найважливіший напрямок для ShiftTech, особливо Volkswagen / Audi / Skoda / Seat / USA 2.5.



### Пріоритет 2



Jatco CVT:



- JF011E

- JF015E

- JF016E

- JF017E



Причина: сильна спеціалізація, багато запитів, люди часто шукають по авто, а не по коду коробки.



### Пріоритет 3



DSG та інші коробки — після побудови ядра.



---



## 5. Основні типи записів



База знань повинна містити такі типи записів:



1. Transmission Page — сторінка коробки передач

2. Vehicle Page — сторінка авто

3. Symptom Page — сторінка симптому

4. Error Code Page — сторінка помилки

5. Repair Case Page — сторінка кейсу ремонту

6. Part Page — сторінка деталі

7. FAQ Page — питання-відповідь

8. Diagnostic Guide — діагностичний алгоритм

9. Article — технічна стаття

10. Source Note — нотатка з джерела знань



---



## 6. Transmission Page



Кожна сторінка коробки передач повинна містити:



- назву коробки;

- альтернативні назви;

- виробника;

- тип коробки;

- роки використання;

- автомобілі, на які встановлювалась;

- двигуни, з якими зустрічається;

- типові симптоми;

- типові помилки;

- слабкі місця;

- типові деталі, які виходять з ладу;

- типові ремонти;

- діагностичні рекомендації;

- пов'язані кейси ShiftTech;

- FAQ;

- посилання на пов'язані авто, симптоми, помилки та кейси.



---



## 7. Vehicle Page



Кожна сторінка автомобіля повинна містити:



- марку;

- модель;

- покоління;

- роки випуску;

- двигун;

- ринок;

- можливі коробки;

- як клієнту зрозуміти, яка коробка стоїть;

- типові симптоми;

- типові помилки;

- типові причини поломок;

- пов'язані кейси;

- посилання на сторінку коробки передач.



Ключовий принцип: клієнт часто шукає не код коробки, а своє авто.



---



## 8. Symptom Page



Кожна сторінка симптому повинна містити:



- опис симптому простою мовою;

- як клієнт це відчуває;

- на яких авто зустрічається;

- на яких коробках зустрічається;

- можливі причини;

- що небезпечно;

- чи можна їхати далі;

- що перевірити спочатку;

- пов'язані помилки;

- пов'язані кейси;

- рекомендацію на діагностику.



---



## 9. Error Code Page



Кожна сторінка помилки повинна містити:



- код помилки;

- розшифровку;

- просте пояснення для клієнта;

- технічне пояснення для майстра;

- коробки, на яких зустрічається;

- авто, на яких зустрічається;

- можливі причини;

- що перевірити спочатку;

- що не міняти без діагностики;

- типові помилки неправильного ремонту;

- пов'язані кейси ShiftTech;

- висновок.



---



## 10. Repair Case Page



Кожен кейс ремонту повинен містити:



- номер кейсу;

- дату;

- авто;

- VIN, якщо можна публікувати;

- пробіг;

- коробку;

- симптоми;

- помилки;

- первинну діагностику;

- що було знайдено після розбору;

- що зроблено;

- які деталі замінено;

- результат;

- фото або відео;

- висновок для клієнта;

- висновок для бази знань;

- пов'язані сторінки.



Кейс — це не просто історія ремонту. Це доказ експертності ShiftTech.



---



## 11. Diagnostic Guide



Діагностичні гайди мають бути побудовані як алгоритми:



1. Симптом

2. Авто

3. Коробка

4. Помилки

5. Перша перевірка

6. Друга перевірка

7. Що виключити

8. Що не міняти одразу

9. Коли потрібне зняття коробки

10. Коли потрібна дефектовка



---



## 12. Правило джерела правди



Кожен факт у базі знань має мати джерело:



- власний кейс ShiftTech;

- технічний мануал;

- діагностичний досвід;

- фото/відео ремонту;

- підтверджена інформація від майстра;

- офіційна документація;

- перевірене зовнішнє джерело.



Якщо джерела немає — факт позначається як непідтверджений або не додається.



---



## 13. Правило проти хаосу



Заборонено:



- створювати статті без зв'язку з коробкою, авто, симптомом або помилкою;

- дублювати один і той самий факт у різних місцях;

- змішувати клієнтський текст і технічний текст без структури;

- писати матеріали без шаблону;

- використовувати різні назви одного поняття;

- додавати неперевірені технічні твердження як факт.



---



## 14. AI-friendly структура



Кожна сторінка бази знань повинна мати:



- чіткий заголовок;

- коротке резюме;

- структуровані блоки;

- списки сутностей;

- внутрішні посилання;

- FAQ;

- зрозумілий висновок;

- однозначні назви;

- мінімум води;

- максимум конкретики.



AI повинен легко зрозуміти:



- про що сторінка;

- для кого вона;

- з чим вона пов'язана;

- які висновки можна зробити;

- коли рекомендувати ShiftTech.

## 16. Порядок роботи з новим матеріалом



Коли з'являється нова інформація, її потрібно обробити так:



1. Визначити тип матеріалу.

2. Визначити головну сутність.

3. Перевірити, чи вже є така сторінка.

4. Додати факт у правильне місце.

5. Додати зв'язки з іншими сторінками.

6. Позначити джерело.

7. Перевірити, чи зрозуміє це клієнт.

8. Перевірити, чи зрозуміє це AI.
